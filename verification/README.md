# Pattern Adoption Verification

Derived data produced by the `/verify-patterns` pipeline. Everything in this directory is
regenerated by pipeline runs — never hand-edit. `patterns.yaml` at the repository root remains
the canonical pattern registry.

## How it works

1. `patterns.yaml` plus the experiments reference table define the catalog — the set of
   pattern slugs the pipeline must account for.
2. Each `/verify-patterns` run picks the stalest patterns and searches the web three ways:
   by **name**, by **mechanism** (the behavior, name excluded), and by **artifact**
   (GitHub code-fingerprint search).
3. An independent verify pass re-fetches every candidate URL; only sources that survive are
   written to `evidence/<slug>.yaml`, each with a tier (T1–T5, how substantial the source is)
   and a `match` value (whether the source uses our name for the practice).
4. `scripts/validate-evidence.py` recomputes `adoption_score`, `verdict`, and
   `naming_alignment` from the entries. CI rejects hand-asserted values, orphaned evidence
   files, and catalog patterns with no evidence and no allowlist entry.
5. Results flow out as per-pattern evidence PRs, one batched README source-line PR,
   `needs-evidence` issues, and discovery candidates. Renames and demotions stay human
   decisions — the pipeline only supplies evidence.

## What this answers

Each file in `evidence/` replaces "I think this pattern is real" with tiered, dated, citable
evidence. Adoption is measured at the level of **practice, not terminology**: a pattern widely
practiced under a different name — or no name — still counts as adopted. Two separate questions
are answered per pattern:

1. **Is the practice established in industry?** → `adoption_score` and `verdict`
2. **Does the industry use our name for it?** → `naming_alignment` and `terminology_variants`

These deliberately never collapse into one signal: `verified` with `naming_alignment: none`
means "real practice, not our name for it" (an alias/rename discussion), while `unverified` means
"the practice itself may not be established" (a demotion discussion). Both discussions are
human decisions — the pipeline only supplies evidence.

## Evidence tiers

Every admitted source is scored against this rubric. At most 3 entries per tier are admitted —
the validator rejects a file that exceeds the cap, so keep the 3 strongest and drop the rest.

| Tier | Evidence type | Weight | Admission test |
|------|--------------|--------|----------------|
| T1 | Shipped product/tooling implementing the pattern | 5 | Can a user run it today? |
| T2 | Vendor official docs or reference architecture | 4 | Canonical docs, not a blog |
| T3 | Conference talk or peer-reviewed paper | 3 | Named speaker, dated event, recording/PDF exists |
| T4 | Practitioner blog post with implementation detail | 2 | Code or config shown, not opinion |
| T5 | Social discussion, opinion post, podcast mention | 1 | Named person, dated, linkable |

This table is the single human-readable statement of the rubric; `scripts/validate-evidence.py`
is the enforcing implementation. The `/verify-patterns` command references both rather than
restating them.

## Reading an evidence file

**`last_checked`** — the date the pipeline last ran this pattern's searches and re-verified its
sources. Files written before the rename may still call this field `verified`; the validator
accepts the legacy name until they are next regenerated.

**`date`** (per evidence entry) — the source's own publication or last-update date when it
states one; omitted entirely for undated pages. `retrieved` is always the date the pipeline
fetched the URL.

**`adoption_score`** — sum of tier weights across accepted entries (at most 3 admitted per tier), so the
range is 0–45. Rough bands: **0–2** no meaningful evidence · **3–7** thin signal · **8+ with at
least one T1–T3 entry** established practice, not just blog chatter. Example:
`24 = 3×T1 (15) + 1×T3 (3) + 3×T4 (6)` means three shipped tools implement the practice, one
conference talk presents it, and three practitioner posts show working implementations.

**`verdict`** — computed by `scripts/validate-evidence.py`, never asserted:

- `verified` — score ≥ 8 AND ≥ 1 entry from T1–T3. The practice demonstrably exists in runnable
  tools, canonical vendor docs, or the conference/paper record.
- `weak` — everything in between, including high scores built only on T4/T5 sources.
- `unverified` — score ≤ 2 after all three search modes ran. Routed to a `needs-evidence`
  GitHub issue for human review, never auto-demoted.

**`naming_alignment`** — computed from each entry's `match` field
(`named` = source uses our pattern name · `aliased` = source uses a different stable name for
the same practice · `unnamed` = source practices the mechanism without naming it):

- `strong` — more than half of entries are `named`.
- `weak` — some `named`/`aliased` entries, but most sources call the practice something else.
- `none` — zero `named` or `aliased` entries.

**`terminology_variants`** — the names other sources use for the same practice. These feed the
next run's name-mode search and any alias/rename discussion.

**`mechanism_quote`** — required on every `aliased`/`unnamed` entry: a verbatim sentence from
the source demonstrating the mechanism, so "same practice" claims are checkable rather than
vibes-based analogies.

## How evidence is collected and trusted

- Three search modes per pattern, ≤ 12 queries total: **name** (pattern name + synonyms),
  **mechanism** (the behavior searched with the name excluded), and **artifact** (GitHub code
  search for the pattern's config/code fingerprint).
- Producer ≠ grader: an independent verify pass re-fetches every URL before it is written and
  must confirm the page exists and supports the claim. For `aliased`/`unnamed` entries the
  grader must confirm same problem, same mechanism, same outcome — all three — or reject.
  Default is reject.
- A URL only appears in these files if it was fetched during the run that wrote it. Every entry
  carries the `retrieved` date.
- `scripts/validate-evidence.py` (enforced in CI via `tests/test_evidence_files.py`) recomputes
  score, verdict, and naming alignment on every file — a hand-asserted verdict fails validation.

## Files

- `evidence/<slug>.yaml` — one file per pattern, keyed by the pattern's README anchor slug.
- `pending-evidence.yaml` — slugs allowed to lack an evidence file. The validator's
  `--registry` cross-check fails CI when a catalog pattern has neither an evidence file nor an
  entry here (and when an evidence file's slug no longer exists in the catalog), so adding,
  renaming, or removing a pattern forces this directory to move with it. When a pattern gains
  evidence, remove its slug — the validator prints a NOTE until you do.
- `pattern-inventory.yaml` — ephemeral per-run working file (gitignored, never committed);
  each pipeline run regenerates it to prioritize the stalest patterns.
