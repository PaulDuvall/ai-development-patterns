# Pattern Verification & Discovery Pipeline

Replace "I think this pattern is real" with tiered, dated, citable schema-v2 evidence. Every
verified adoption claim must trace to an auditable search, an independently owned source, fetched
content, and a verifier run. Adoption is measured at the level of **practice, not terminology**:
a pattern widely practiced under a different name — or no name — still counts as adopted.

Arguments: $ARGUMENTS
- `--pattern <name>`: Phases 2-3 for one pattern. Validate the name against `patterns.yaml` first; fail fast if unknown.
- `--discover-only`: Phase 4 only.
- `--full`: all phases, all patterns.
- Default (no args): Phases 1-3 on up to 10 patterns whose evidence is missing, has
  `legacy-import` provenance, or is older than `--stale-days`, excluding slugs already changed by
  a trusted generated verification PR. If fewer than 10 qualify, verify only those — possibly
  zero; an idle run is correct, not a failure.
- `--stale-days <n>` (default 90): staleness threshold for prioritization.
- `--since <days>` (default 30): lookback window for Phase 4.

## Ground rules (non-negotiable)

1. Read `pattern-spec.md`, `CLAUDE.md`, and `verification/README.md` (the rubric and evidence
   schema) before writing anything. All edits must conform.
2. **`patterns.yaml` is the canonical pattern registry** (consumed by `tests/`, the badge script, and
   the site generator). Never create a competing hand-maintained registry; everything under
   `verification/` is derived data, regenerated by this command.
3. Get today's date from `date -I`, never from memory. All dates ISO 8601 in YAML; README source
   lines follow the existing long-form date style. Every evidence entry carries `retrieved:`.
4. Never fabricate URLs, dates, authors, queries, candidate counts, organizations, quotes, hashes,
   or verifier metadata. **A URL may only be written after it has been fetched this run and its
   content supports the claim.** `evidence: none found` is valid only when the complete three-mode
   search record proves the search ran.
5. Research and verification subagents receive only a read-scoped repository token. Direct
   `git`/`gh` tools are disallowed, proposed writes are limited to the fixed candidate paths, and a
   separate scoped publisher runs only after clean deterministic validation.
6. Use Python 3.11+. Before any publish, run:
   `python3 scripts/validate-evidence.py --registry patterns.yaml --allowlist verification/pending-evidence.yaml`,
   `python3 scripts/generate-verification-status.py --check`, and
   `python3 -m pytest tests/test_evidence_files.py -m "not slow"`. If README changes, also run the
   full pattern/site generators and `pytest tests/`.
7. Do not rename patterns. Names are strict two-word Title Case (`scripts/validate-pattern-names.py`).
   Record terminology variants as data; renaming is a human decision.
8. Distinguish facts from inference. A blog post claiming adoption is tiered evidence, not truth.
9. Never merge. All PRs await human acceptance (they will trigger `claude-code-review.yml`).
10. Bounds: ≤12 search queries per pattern across all three search modes, stop a pattern's search
    after 2 consecutive queries that surface nothing new, and open one batched PR per run. If any phase fails
    3 times: stop, make no further writes, and record what failed and what was attempted in the
    final report / step summary.

## Phase 0 — Bootstrap (idempotent)

1. Confirm Python 3.11+, `verification/evidence/`, and the pinned `tests/requirements.txt` exist.
2. If `scripts/validate-evidence.py` or `tests/test_evidence_files.py` is missing, **abort the
   run** and tell the human to restore the guard — the pipeline must never regenerate the guard
   that polices its own output. **Verdicts are computed by that script, never asserted by the
   model** (rubric and rules: `verification/README.md`).
3. Under GitHub Actions, consume the workflow-generated inventory/candidate paths; do not replace
   them with model-authored repository writes or dynamically chosen artifact paths.

## Phase 1 — Inventory (deterministic, no web access)

1. Run `scripts/build-verification-inventory.py` with the requested mode, stale threshold, pattern,
   and limit (plus the workflow-provided in-flight slug file when present). This script—not model
   judgment—parses `patterns.yaml` (canonical main names, types, and maturity) and the reference
   table in `experiments/README.md` (experimental patterns, tracked with `location: experimental`).
2. Canonical work-item key = the pattern's README anchor slug (the same id the site generator
   derives). One evidence file per slug — one writer per resource.
3. Regenerate `verification/pattern-inventory.yaml` (an ephemeral working file — gitignored,
   regenerated every run, never committed): name, slug, type, maturity, location, last_checked,
   provenance_status, evidence_count, aliases, and freshness. Treat `legacy-import` as needing
   refresh regardless of its date.
4. **In-flight dedup:** a pattern changed by a same-repository generated branch matching
   `verify/refresh-<run-id>-<attempt>` is already being reviewed. In Actions, the trusted `prepare`
   job verifies those head-repository and branch constraints, queries their file lists, and writes
   the evidence slugs to the script's `--inflight-slugs` input before research starts. Untrusted
   fork PRs cannot starve the worklist. For a local run, create the equivalent input outside the
   research agent.
5. Report: total patterns, zero-evidence patterns, patterns staler than `--stale-days`. This drives
   default prioritization.
6. Use the script-generated worklist (`.verify-worklist` manually or
   `verification/run-plan/worklist.txt` in Actions) as the immutable research scope. The final
   summary must compare selected and completed slugs so a timeout cannot silently truncate coverage.

## Phase 2 — Evidence collection (per pattern; fan out subagents, read-only)

Search each pattern in **three modes** — a pattern practiced without our name for it must still be
findable:

- **(a) Name mode** — pattern name + plausible industry synonyms.
- **(b) Mechanism mode** — 3-5 queries derived from the pattern's Description and implementation
  section with the pattern name **excluded** from the query (search the behavior, not the label —
  e.g. for Security Sandbox: "run coding agent in container without network access"). Also search
  the pattern's problem statement ("how do teams stop AI agents leaking secrets") and record what
  sources call their solution — that feeds `terminology_variants` and improves the next run's
  name-mode search.
- **(c) Artifact mode** — GitHub code search for the pattern's config/code fingerprint, derived from
  the pattern's own code examples (e.g. `network_mode: none` alongside agent tooling, CI YAML
  invoking an LLM reviewer). Many repos doing the thing without naming it is strong adoption
  evidence with zero terminology match.

Source universe: vendor docs (Anthropic, OpenAI, Cursor, GitHub), engineering blogs
(martinfowler.com, company eng blogs), conference talks (AI Engineer, QCon, KubeCon), arxiv,
high-star GitHub repos, podcast transcripts.

Score every admitted hit against the evidence-tier rubric in `verification/README.md`
(**the single human-readable copy** — T1–T5 with weights 5/4/3/2/1, admission tests per tier,
at most 3 entries admitted per tier). Record what the source demonstrates; do not editorialize.

**Independent verify pass (producer ≠ grader).** A separate read-only subagent re-fetches every
URL before it is written and confirms the page exists and supports the recorded claim. The bar
scales with how the source relates to our name:

- `named` entries: page live, claim supported. Default reject on doubt.
- `aliased` / `unnamed` entries: the risk is false analogy, not fabrication. The grader gets the
  pattern's full README section plus the source and must confirm **same problem, same mechanism,
  same outcome — all three** — citing a verbatim `mechanism_quote` from the source that demonstrates
  the mechanism. Topical similarity is a reject. Two of three is a reject. Default reject.

Only entries surviving verification enter the fixed-path candidate artifact. Complete candidates
record the exact search and source provenance required by schema v2:

```yaml
schema_version: 2
provenance_status: complete
pattern: Spec-Driven Development
slug: spec-driven-development
last_checked: 2026-07-10
search:
  run_id: "github-actions:123456789"
  run_url: https://github.com/PaulDuvall/ai-development-patterns/actions/runs/123456789
  checked_at: 2026-07-10
  modes:
    name:
      queries: ["spec driven development coding agents"]
      candidate_count: 4
    mechanism:
      queries: ["written requirements generate implementation tasks coding agent"]
      candidate_count: 6
    artifact:
      queries: ["path:.specify/memory/constitution.md"]
      candidate_count: 3
adoption_score: 5
tier_counts: {T1: 1}
adoption_dimensions:
  implementation_available: true
  independent_adoption: false
  independence_groups: 1
naming_alignment: strong
terminology_variants: []
evidence:
  - tier: T1
    match: named
    mechanism_quote: "Specifications become executable and generate working implementations."
    source: GitHub Spec Kit
    source_kind: open_source_implementation
    organization: GitHub
    independence_group: github
    url: https://github.com/github/spec-kit
    resolved_url: https://github.com/github/spec-kit
    content_sha256: 0123456789abcdef0123456789abcdef0123456789abcdef0123456789abcdef
    verifier:
      method: automated
      model: example-model
      prompt_version: evidence-v2
      run_url: https://github.com/PaulDuvall/ai-development-patterns/actions/runs/123456789
    retrieved: 2026-07-10
    claim: Runnable toolkit implements a specification-to-plan-to-tasks workflow.
verdict: weak
```

Entry `date` is the source's own publication/update date — omit the field for undated pages
(never copy the retrieval date into it; `retrieved` already records the fetch).

Each admitted entry needs a unique `independence_group` within the file and a unique canonical
`resolved_url` across the evidence set. T3 must be a recorded conference talk or genuinely
peer-reviewed publication; an arXiv preprint alone does not qualify. Use
`python3 scripts/evidence_content.py <url> --quote <exact-quote>` to follow redirects, normalize
visible response text, compute SHA-256, and prove the quote is present. The displayed score includes T5, but the verified
threshold counts T1–T4 points only; T5 may inform discovery/naming but never supplies a deciding
adoption point.

Run the candidate through `scripts/validate-evidence.py` in a clean checkout before the publisher
receives it. The model never writes a `verified` assertion that the deterministic recomputation does
not produce.

**Verdict and naming-alignment rules are computed by `scripts/validate-evidence.py`, never
asserted — the human-readable rules live in `verification/README.md` ("Reading an evidence
file").**

Verdict and naming alignment answer different questions and drive different follow-ups:
`verified` + `naming_alignment: none` means "the practice is real but nobody uses our name" — an
alias/rename discussion. `unverified` after all three search modes means "the practice itself may
not be established" — a demotion discussion. Never collapse the two.

## Phase 3 — Deterministic publish (one batched PR)

The research process may edit only the candidate paths in its disposable checkout; it never
mutates the remote repository. After research finishes, a clean job validates the fixed-path
candidate artifact, and only then hands that artifact to the scoped publisher.

1. Start one branch from the captured base revision:
   `verify/refresh-<github-run-id>-<run-attempt>`. Never reuse or force-push a run branch.
2. Apply all accepted candidate files. For each new evidence file, remove the same slug from
   `verification/pending-evidence.yaml` in this exact change. Do not publish a temporary overlap.
3. Refresh evidence-derived columns in `verification/DECISIONS.md`; never edit its human-owned
   Decision column.
4. If top evidence changed, refresh at most the top 1–2 README source lines using the existing
   citation convention. If a terminology variant dominates, add an alias mention; never rename.
5. Regenerate shared artifacts once:
   `python3 scripts/generate-verification-status.py`, plus the pattern count/site data generators
   when README changed.
6. Validate the resulting tree in a clean checkout:
   `python3 scripts/validate-evidence.py --registry patterns.yaml --allowlist verification/pending-evidence.yaml`,
   `python3 scripts/generate-verification-status.py --check`, and
   `python3 -m pytest tests/test_evidence_files.py -m "not slow"`, plus
   `EVIDENCE_HASH_STRICT=1 python3 -m pytest tests/test_evidence_content.py -m slow` for complete
   candidates; run `pytest tests/` when README or
   site artifacts changed. Any failure aborts publishing with no repository mutation.
7. Create one lowercase conventional commit and one draft PR containing the evidence, atomic pending-list
   changes, ledger data, status, citations, and any discovery queue updates from this run. Report a
   no-op without a branch when nothing changed.

The publisher prefers a short-lived GitHub App token (`VERIFY_PATTERNS_APP_ID` repository variable
plus `VERIFY_PATTERNS_APP_PRIVATE_KEY` secret). After opening the draft, it explicitly dispatches
`trusted-evidence-validation.yml` on the default branch with the PR number. Trusted base-branch
code then posts the required result on the candidate revision even if the automation-created PR
event remains behind GitHub's approval gate. Never give a mutation credential to the research job.

A `weak` or `unverified` verdict on a non-experimental pattern does not delete or demote it. After
the batch passes and publishes, open or update `Adoption evidence gap: <Pattern Name>` with label
`needs-evidence`, linking the immutable search run. A candidate that computes to `verified` only
closes that issue after its draft PR is accepted and merged. Demotion remains a human decision.

For each `verified` pattern with `naming_alignment: none`, `weak`, or `aliased`, produce a **naming
recommendation**, not just a flag:

1. Identify the dominant industry term(s) from `terminology_variants` — dominance means multiple
   independent (unaffiliated) sources, weighted by evidence tier.
2. Derive candidate names from the dominant term and run each through the pattern-spec.md naming
   validation checklist: exactly two words Title Case (hyphenated compounds count as one word),
   Noun+Noun or Adj+Noun, no "AI" prefix, domain-specific vocabulary, 2-4 syllables per word,
   unique across the catalog including experiments, passes the "Use the X Y pattern to..." test,
   and describes a principle rather than a single vendor's feature name.
3. Recommend exactly one action, with rationale citing specific evidence entries by source:
   - **keep** — current name is defensible; no spec-compliant, evidence-backed candidate beats it.
   - **keep + alias** — record the industry term via the batched alias mention.
   - **rename to <Candidate>** — only when the candidate passes the full checklist AND either the
     industry term dominates across independent sources or the current name has a documented
     collision/discoverability problem (e.g. name-mode searches drown in an unrelated meaning).
4. Record the recommendation in the final report, and refresh the pattern's row in
   `verification/DECISIONS.md` in the batched PR — data columns only (Alignment,
   Industry terms, Recommendation); the Decision column is human-owned and must never be
   written by the pipeline. Renaming remains a human decision — never open a rename PR; if a
   human accepts a rename, PATTERN_MIGRATION_GUIDE.md governs the mechanics (anchors, slugs,
   evidence file key, site regeneration).

## Phase 4 — Unknown-unknowns discovery

1. Scan the last `--since` days: arxiv cs.SE/cs.AI, Anthropic/OpenAI/Cursor/GitHub changelogs and
   eng blogs, martinfowler.com, AI Engineer talk titles, trending GitHub repos tagged ai + agents,
   HN front-page agentic-development threads.
2. Extract candidates of two kinds:
   - **Named techniques** — recurring named workflow nouns and control-loop concepts used by
     multiple unaffiliated practitioners for the same repeatable technique.
   - **Unnamed practices** — a workflow described consistently across independent sources with no
     stable name. These are the strongest candidates: a naming vacuum the catalog can fill.
3. Diff against inventory names + aliases + terminology_variants (match on mechanism, not just
   string — an unnamed candidate may be an existing pattern in disguise; if so, route it to that
   pattern's evidence file instead). Unmatched = candidate.
4. For each candidate with ≥2 independent sources: append an entry to `experiments/NOTES.md` under
   the existing `## Pattern Exploration Queue`, using its established fields (Status: Early
   exploration, Date Added, Description, Related Patterns, Research Questions) plus the sources and
   a proposed two-word Title Case name per pattern-spec.md naming rules. Do not create full
   patterns — NOTES.md is the intake gate.
5. In a full run, include queue changes in the Phase 3 batched PR. In `--discover-only`, open one
   `discover/<yyyymmdd>-<run-id>` PR with commit
   `docs(experiments): queue <n> discovered pattern candidates`.

## Final report (never committed)

Table: pattern | prior verdict | new verdict | score delta | naming_alignment | PR/issue link.
Then the Phase 3 naming recommendations, one row per `verified` pattern with non-strong naming
alignment: pattern | dominant industry term | spec-checked candidate | recommendation (keep /
keep + alias / rename to X) | evidence basis. Show why rejected candidates failed the checklist.
Then discovery candidates: term (or "unnamed" + proposed name) | sources | closest existing pattern.
End with the three highest-leverage follow-ups, keeping naming recommendations (`verified` but weak/no
naming alignment) separate from evidence gaps (`unverified`). Note anything skipped due to bounds
(query caps, pattern selection, or time/turn limits) — silent truncation reads as full coverage.

When running under GitHub Actions (`GITHUB_ACTIONS` env var is set), also append the full report as
markdown to the file named by `$GITHUB_STEP_SUMMARY` so it renders on the run's summary page.
