# Pattern Verification & Discovery Pipeline

Replace "I think this pattern is real" with tiered, dated, citable evidence. Every adoption claim
traces to a source the same way a spec traces to a test. Adoption is measured at the level of
**practice, not terminology**: a pattern widely practiced under a different name — or no name —
still counts as adopted, and the pipeline must be able to find it.

Arguments: $ARGUMENTS
- `--pattern <name>`: Phases 2-3 for one pattern. Validate the name against `patterns.yaml` first; fail fast if unknown.
- `--discover-only`: Phase 4 only.
- `--full`: all phases, all patterns.
- Default (no args): Phases 1-3 on the 10 patterns with the oldest or missing evidence.
- `--stale-days <n>` (default 90): staleness threshold for prioritization.
- `--since <days>` (default 30): lookback window for Phase 4.

## Ground rules (non-negotiable)

1. Read `pattern-spec.md` and `CLAUDE.md` before writing anything. All edits must conform.
2. **`patterns.yaml` is the canonical pattern registry** (consumed by `tests/`, the badge script, and
   the site generator). Never create a competing hand-maintained registry; everything under
   `verification/` is derived data, regenerated by this command.
3. Get today's date from `date -I`, never from memory. All dates ISO 8601 in YAML; README source
   lines follow the existing long-form date style. Every evidence entry carries `retrieved:`.
4. Never fabricate URLs, dates, authors, or quotes. **A URL may only be written after it has been
   fetched this run and its content supports the claim.** If searching yields nothing, record
   `evidence: none found` with the retrieved date.
5. Research and verification subagents are read-only — they never touch git or repo files. Only the
   top-level session writes.
6. Validation before any commit — the same checks CI runs (`.github/workflows/pattern-validation.yml`):
   `python3 scripts/update-pattern-count.py && python3 scripts/generate-patterns-data.py && pytest tests/`
   CI fails if site artifacts are not regenerated after README pattern edits. Length guideline:
   patterns target 1,200-2,400 words, hard max 3,500 (pattern-spec.md).
7. Do not rename patterns. Names are strict two-word Title Case (`scripts/validate-pattern-names.py`).
   Record terminology variants as data; renaming is a human decision.
8. Distinguish facts from inference. A blog post claiming adoption is tiered evidence, not truth.
9. Never merge. All PRs await human acceptance (they will trigger `claude-code-review.yml`).
10. Bounds: ≤12 search queries per pattern across all three search modes, stop a pattern's search
    after 2 consecutive queries that surface nothing new, ≤12 open PRs per run. If any phase fails
    3 times, stop and report per the failure protocol.

## Phase 0 — Bootstrap (idempotent)

1. `mkdir -p verification/evidence`
2. `gh label create needs-evidence --description "Pattern lacks adoption evidence" --color D93F0B 2>/dev/null || true`
3. If `scripts/validate-evidence.py` does not exist, create it plus `tests/test_evidence_files.py`
   in a one-time PR (`chore(verification): add evidence schema guard`). The guard validates every
   `verification/evidence/*.yaml`: required fields present, ISO dates, ≤3 entries per tier,
   `match` is a valid enum value, `mechanism_quote` present on every aliased/unnamed entry,
   `adoption_score` equals the recomputed sum, `verdict` and `naming_alignment` equal their
   recomputed values. **Verdicts are computed by this script, never asserted by the model.**

## Phase 1 — Inventory (deterministic, no web access)

1. Parse `patterns.yaml` (canonical names/maturity/descriptions), the Complete Pattern Reference
   table in `README.md` (Type: Foundation | Development | Operations | Workflow; Dependencies), and
   `experiments/README.md` (experimental patterns, tracked with `location: experiments`).
2. Canonical work-item key = the pattern's README anchor slug (the same id the site generator
   derives). One evidence file per slug — one writer per resource.
3. Regenerate `verification/pattern-inventory.yaml` (a derived artifact — never hand-edit):
   name, slug, type, maturity, location, last_verified (from `verification/evidence/<slug>.yaml`),
   evidence_count, aliases.
4. Report: total patterns, zero-evidence patterns, patterns staler than `--stale-days`. This drives
   default prioritization.

## Phase 2 — Evidence collection (per pattern; fan out subagents, read-only)

Search each pattern in **three modes** — a pattern practiced without our name for it must still be
findable:

- **(a) Name mode** — pattern name + plausible industry synonyms.
- **(b) Mechanism mode** — 3-5 queries derived from the pattern's Description and implementation
  section with the pattern name **excluded** from the query (search the behavior, not the label —
  e.g. for Security Sandbox: "run coding agent in container without network access"). Also search
  the pattern's problem statement ("how do teams stop AI agents leaking secrets") and record what
  sources call their solution — that feeds `terminology_variants` and improves the next run's
  name-mode search.
- **(c) Artifact mode** — GitHub code search for the pattern's config/code fingerprint, derived from
  the pattern's own code examples (e.g. `network_mode: none` alongside agent tooling, CI YAML
  invoking an LLM reviewer). Many repos doing the thing without naming it is strong adoption
  evidence with zero terminology match.

Source universe: vendor docs (Anthropic, OpenAI, Cursor, GitHub), engineering blogs
(martinfowler.com, company eng blogs), conference talks (AI Engineer, QCon, KubeCon), arxiv,
high-star GitHub repos, podcast transcripts.

Score every admitted hit against this rubric. Record what the source demonstrates; do not editorialize.

| Tier | Evidence type | Weight | Test |
|------|--------------|--------|------|
| T1 | Shipped product/tooling implementing the pattern | 5 | Can a user run it today? |
| T2 | Vendor official docs or reference architecture | 4 | Canonical docs, not a blog |
| T3 | Conference talk or peer-reviewed paper | 3 | Named speaker, dated event, recording/PDF exists |
| T4 | Practitioner blog post with implementation detail | 2 | Code or config shown, not opinion |
| T5 | Social discussion, opinion post, podcast mention | 1 | Named person, dated, linkable |

**Independent verify pass (producer ≠ grader).** A separate read-only subagent re-fetches every
URL before it is written and confirms the page exists and supports the recorded claim. The bar
scales with how the source relates to our name:

- `named` entries: page live, claim supported. Default reject on doubt.
- `aliased` / `unnamed` entries: the risk is false analogy, not fabrication. The grader gets the
  pattern's full README section plus the source and must confirm **same problem, same mechanism,
  same outcome — all three** — citing a verbatim `mechanism_quote` from the source that demonstrates
  the mechanism. Topical similarity is a reject. Two of three is a reject. Default reject.

Only entries surviving verification are written to `verification/evidence/<slug>.yaml`:

```yaml
pattern: Spec-Driven Development
slug: spec-driven-development
verified: 2026-07-02
adoption_score: 7         # computed: 1×T1 (5) + 1×T4 (2); at most 3 entries admitted per tier
naming_alignment: weak    # computed: 1 of 2 entries named — strong needs a named majority
terminology_variants:
  - term: "spec-first development"
    used_by: "GitHub (Spec Kit docs)"
    url: https://...
evidence:
  - tier: T1
    match: named          # named | aliased | unnamed
    source: "GitHub Spec Kit"
    url: https://...
    date: 2025-09-02
    retrieved: 2026-07-02
    claim: "Toolkit implementing spec-to-plan-to-tasks workflow"
  - tier: T4
    match: unnamed
    mechanism_quote: "we write the full spec in markdown, review it, then have the agent generate the code from it"
    source: "..."
    url: https://...
    date: 2026-01-15
    retrieved: 2026-07-02
    claim: "Team practices spec-first agent workflow without naming it"
verdict: weak             # computed — never asserted (score 7 < 8, so not verified)
```

Then run `scripts/validate-evidence.py` to confirm score, verdict, and naming_alignment recompute
cleanly.

**Verdict rules (computed, exhaustive):**
- `verified` = adoption_score ≥ 8 AND ≥1 T1-T3 entry.
- `unverified` = adoption_score ≤ 2.
- `weak` = everything else (including high scores built only on T4/T5).

**Naming alignment (computed, exhaustive):**
- `strong` = more than half of evidence entries are `named`.
- `none` = zero `named` or `aliased` entries.
- `weak` = everything else.

Verdict and naming alignment answer different questions and drive different follow-ups:
`verified` + `naming_alignment: none` means "the practice is real but nobody uses our name" — an
alias/rename discussion. `unverified` after all three search modes means "the practice itself may
not be established" — a demotion discussion. Never collapse the two.

## Phase 3 — Repo updates (writes split by conflict domain)

All branches from up-to-date `origin/main`. Commit style matches the repo: lowercase conventional
commits, no Co-Authored-By trailers. If a branch for the same slug and date already exists (a
same-day re-run), reset it to `origin/main`, redo the change, and `git push --force-with-lease`
rather than stacking commits or inventing a variant branch name.

**(a) Per-pattern evidence PRs** — branch `verify/<slug>-<yyyymmdd>`, touching ONLY
`verification/evidence/<slug>.yaml`. These files are disjoint; open freely in parallel.
Commit: `docs(verification): verify <slug> — <verdict>, score <n>`. PR body = evidence summary as
markdown. Only open a PR when the verdict or evidence actually changed.

**(b) One batched README/site PR per run** — branch `verify/sources-<yyyymmdd>`. For patterns whose
top evidence changed, add/refresh the pattern section's source line using the EXISTING convention —
`**Source**: Author, "[Title](url)", Month D, YYYY` — top 1-2 entries only; evidence detail lives in
`verification/`, preserving the 2-3 minute scannability rule. Regenerate site artifacts ONCE here
(`update-pattern-count.py`, `generate-patterns-data.py`), run `pytest tests/`, commit
`docs(patterns): refresh source citations from adoption evidence`.
Rationale: `index.html` and `assets/js/patterns-data.js` are whole-file regenerated — per-pattern
PRs touching them would all conflict.

**(c)** `unverified` verdict on a non-experimental pattern: do NOT delete or demote. Open a GitHub
issue `Adoption evidence gap: <Pattern Name>` with the evidence YAML, label `needs-evidence`. State
in the issue that mechanism and artifact searches were also run, so the gap reflects the practice,
not a search-method failure. Demotion is a human decision.

**(d)** If a terminology variant dominates industry usage: add an alias mention in the pattern
description ("also referred to as X by Y") in the batched PR. Never rename.

**(e)** `verified` with `naming_alignment: none` or `weak`: flag in the final report as a naming
discussion (with the dominant industry term, if one exists). Renaming is a human decision — do not
open a PR for it.

## Phase 4 — Unknown-unknowns discovery

1. Scan the last `--since` days: arxiv cs.SE/cs.AI, Anthropic/OpenAI/Cursor/GitHub changelogs and
   eng blogs, martinfowler.com, AI Engineer talk titles, trending GitHub repos tagged ai + agents,
   HN front-page agentic-development threads.
2. Extract candidates of two kinds:
   - **Named techniques** — recurring named workflow nouns and control-loop concepts used by
     multiple unaffiliated practitioners for the same repeatable technique.
   - **Unnamed practices** — a workflow described consistently across independent sources with no
     stable name. These are the strongest candidates: a naming vacuum the catalog can fill.
3. Diff against inventory names + aliases + terminology_variants (match on mechanism, not just
   string — an unnamed candidate may be an existing pattern in disguise; if so, route it to that
   pattern's evidence file instead). Unmatched = candidate.
4. For each candidate with ≥2 independent sources: append an entry to `experiments/NOTES.md` under
   the existing `## Pattern Exploration Queue`, using its established fields (Status: Early
   exploration, Date Added, Description, Related Patterns, Research Questions) plus the sources and
   a proposed two-word Title Case name per pattern-spec.md naming rules. Do not create full
   patterns — NOTES.md is the intake gate.
5. Single PR: branch `discover/<yyyymmdd>`, commit
   `docs(experiments): queue <n> discovered pattern candidates`.

## Final report (never committed)

Table: pattern | prior verdict | new verdict | score delta | naming_alignment | PR/issue link.
Then discovery candidates: term (or "unnamed" + proposed name) | sources | closest existing pattern.
End with the three highest-leverage follow-ups, keeping naming discussions (`verified` but weak/no
naming alignment) separate from evidence gaps (`unverified`). Note anything skipped due to bounds
(query caps, PR cap) — silent truncation reads as full coverage.

When running under GitHub Actions (`GITHUB_ACTIONS` env var is set), also append the full report as
markdown to the file named by `$GITHUB_STEP_SUMMARY` so it renders on the run's summary page.
